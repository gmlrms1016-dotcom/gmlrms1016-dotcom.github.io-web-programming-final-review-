📦 Git 소스 관리 기본 흐름13주 · 핵심

작업 디렉토리 → 스테이징 영역(Index) → 지역(로컬) 저장소 → 원격 저장소 4단계로 흐릅니다. 각 단계 사이를 옮기는 명령어를 외우는 게 핵심이에요.

흐름명령어설명
작업 → 스테이징git add변경 파일을 커밋 후보(Index)에 올림
스테이징 → 지역git commit -m올린 변경을 하나의 커밋으로 저장
지역 → 원격git push로컬 커밋을 GitHub에 업로드
원격 → 지역git pull원격 변경을 가져와 병합
원격 → 새 로컬git clone원격을 통째로 복제해 새 저장소 생성

⌨️ 주차별 명령어 레퍼런스 (10~14주차)암기

🔁 우리는 git checkout 대신 git switch를 씁니다. 전환은 git switch <브랜치>, 생성+전환은 git switch -c <브랜치> (예전 checkout -b의 -b → switch에서는 -c). 13주차가 시험의 핵심이라 기본 펼침입니다.

되돌리기 — Git Reset

명령어설명
git reset <파일> / git reset스테이지 취소(add 취소) — staged → unstaged
git reset --soft <시점>커밋 취소, staging은 유지
git reset --mixed <시점> (기본값)커밋·staging 취소, 로컬 파일 변경은 유지
git reset --hard <시점>커밋·staging·로컬 변경 모두 취소
시점 표기 HEAD^ / HEAD~3최신 1개 전 / 최근 3개 전 (예 git reset --mixed HEAD^)

# 코드리뷰 후 Merge → "개발 내용 확인, 이상 없음"
git switch main

실습 3 — 개발 브랜치 + Rebase (선형 이력 만들기)

function·main·bugfix 세 브랜치를 만들고, rebase로 한 줄(선형)로 정리한 뒤 fast-forward merge.

(1) 브랜치 3개 생성·커밋
git switch -c function
echo "function 개발" >> code1.txt
git add . && git commit -m "function 개발중"
git switch main
echo "main 개발" >> code1.txt
git add . && git commit -m "main 개발중"
git switch -c bugfix origin/main
echo "bug 수정" >> code2.txt
git add . && git commit -m "bug 수정"

(2) main에 bugfix를 rebase → 선형
git switch main
git rebase bugfix

(3) function을 main 위로 rebase (충돌 시 수정→git add .→git rebase --continue)
git switch function
git rebase main

(4) main에 function merge → fast-forward
git switch main
git merge function
git push origin main
🖥️ 실제 실행 로그 — 단계별 git log --all --oneline --graph (내 Git Bash 캡처)
→ rebase로 분기를 선형으로 펴니, 마지막 병합이 새 커밋 없이 fast-forward로 끝남. (커밋 해시는 내 실제 실행값)
실습 4 — Squash Merge (커밋 5개 → 1개로 압축) 
(1) apple 브랜치에 커밋 5개
git switch -c apple
echo "apple1" >> code1.txt && git add . && git commit -m "apple"
echo "apple2" >> code1.txt && git add . && git commit -m "apple"
# ... apple3, apple4, apple5 동일

(2) main에서 squash merge → 하나로 합쳐 새 커밋
git switch main
git merge --squash apple
git commit -m "five apples"     # 5개가 1개 커밋으로

(3) 원격에 push
git push origin main
🖥️ 실제 실행 로그 — squash 전·후 git log --all --oneline --graph (내 Git Bash 캡처)
→ squash로 자잘한 apple 커밋 5개가 “five apples” 한 커밋(13a7d92)으로 합쳐짐. (커밋 해시는 내 실제 실행값)

🔑 핵심 개념 — merge · rebase · squash · PR비교

개념한 줄 정리
fast-forward merge브랜치가 일직선상에 있을 때 포인터만 앞으로 이동하는 병합 (새 커밋 안 생김)
merge다른 브랜치를 현재 브랜치에 합침. 갈래가 있으면 병합 커밋이 생기고 이력이 보존됨
rebase브랜치의 기반(base)을 다른 커밋으로 옮겨 선형 이력 생성 (깔끔하지만 이력을 재작성)
merge --squash여러 커밋을 하나의 커밋으로 압축해 병합 (별도 commit 필요)
Pull Request(PR)원격에서 브랜치 병합을 요청 + 코드리뷰를 진행하는 GitHub 협업 기능
🧠 rebase vs merge : 둘 다 합치지만 — merge는 갈래를 남기고(이력 보존), rebase는 한 줄로 펴서(이력 깔끔) 만든다. squash는 “여러 커밋 → 1개”.

보기2 (목표)
* a1b2c3d (HEAD -> feature, main, origin/main) 첫 커밋
정답 B — 새 브랜치 생성+전환은 git switch -c feature. git branch feature는 만들기만 하고 HEAD는 main에 남아(보기2처럼 HEAD->feature가 안 됨), git switch feature는 없는 브랜치라 오류.

2아래 [보기1]에서 [보기2]로 만들려면? (현재 main, feature가 일직선상 앞에 있음)

보기1 (현재)
* f3e2d1c (feature) 기능 완성
* a1b2c3d (HEAD -> main) 시작
➜명령어?
보기2 (목표)
* f3e2d1c (HEAD -> main, feature) 기능 완성
* a1b2c3d 시작
정답 A — 일직선이라 main 포인터만 feature까지 앞으로 이동 = Fast-Forward 병합. 새 병합 커밋 없이 main이 feature 커밋을 가리킴.

3아래 [보기1]에서 [보기2]로 만들려면? (현재 main, test와 갈라짐 → 병합 커밋 생성)

보기1 (현재)
* aa11aa1 (HEAD -> main) main 수정
| * bb22bb2 (test) test 수정
|/
* cc00cc0 공통 조상
➜명령어?
보기2 (목표)
*   mm33mm3 (HEAD -> main) Merge branch 'test'
|\
| * bb22bb2 (test) test 수정
* | aa11aa1 main 수정
|/
* cc00cc0 공통 조상
정답 A — 양쪽이 갈라진 상태에서 git merge test → 병합 커밋(mm33mm3) 생성 = Three-way 병합. (Fast-Forward와 달리 새 커밋이 생기고 갈래가 합쳐짐)

4아래 [보기1]에서 [보기2]로 만들려면? (갈라진 이력 → 한 줄(선형), 해시도 바뀜)

보기1 (현재)
* c1c1c1c (HEAD -> feature) feature 작업
| * b2b2b2b (main) main 작업
|/
* a0a0a0a 공통 조상
➜명령어?
보기2 (목표)
* d3d3d3d (HEAD -> feature) feature 작업
* b2b2b2b (main) main 작업
* a0a0a0a 공통 조상
정답 B — 갈래를 main 끝으로 옮겨 한 줄(선형)로 = git rebase main. feature 커밋 해시가 c1c1c1c→d3d3d3d로 바뀐 것도 rebase(이력 재작성) 단서. merge였다면 갈래+병합커밋이 남음.

5아래 [보기1]에서 [보기2]로 만들려면? (apple 커밋 5개 → main에 1개로 압축)

보기1 (현재)
* 5a5a5a5 (apple) apple5
* 4b4b4b4 apple4
* 3c3c3c3 apple3
* 2d2d2d2 apple2
* 1e1e1e1 apple1
* 0f0f0f0 (HEAD -> main) base
➜명령어?
보기2 (목표)
* 9a0b1c2 (HEAD -> main) five apples
| * 5a5a5a5 (apple) apple5
| * 4b4b4b4 apple4
| * 3c3c3c3 apple3
| * 2d2d2d2 apple2
| * 1e1e1e1 apple1
|/
* 0f0f0f0 base
정답 C — apple의 커밋 5개가 main에 "five apples" 1개로 압축 = git merge --squash. squash는 자동 커밋이 안 되므로 별도 git commit이 꼭 필요. (apple 브랜치 5개는 옆 갈래로 그대로 남음)

6아래 [보기1]에서 [보기2]로 만들려면? (로컬 main이 origin/main보다 앞섬)

보기1 (현재)
* 7a7a7a7 (HEAD -> main) 새 작업
* 6b6b6b6 (origin/main) 이전 작업
➜명령어?
보기2 (목표)
* 7a7a7a7 (HEAD -> main, origin/main) 새 작업
* 6b6b6b6 이전 작업
정답 B — origin/main이 로컬 main까지 따라옴 = 로컬 커밋을 원격에 업로드 push. git pull은 반대 방향(원격→로컬).

7아래 [보기1]에서 [보기2]로 만들려면? (feature 라벨이 사라짐)

보기1 (현재)
* e1e1e1e (HEAD -> main, feature) 병합 완료
* d0d0d0d 시작
➜명령어?
보기2 (목표)
* e1e1e1e (HEAD -> main) 병합 완료
* d0d0d0d 시작
정답 A — feature 라벨이 사라짐 = 병합이 끝난 브랜치 삭제 git branch -d feature. (병합 안 된 브랜치를 강제 삭제할 땐 -D)

객관식 — 보기를 클릭하면 정답·해설이 보여요

1작업 디렉토리의 변경 파일을 스테이징 영역에 올리는 명령어는?

정답 B — 작업→스테이징은 git add. 스테이징→지역은 git commit.

2원격 저장소를 복제해 새 로컬 저장소를 만드는 명령어는?

정답 C — git clone <url>. git pull은 기존 저장소에 변경을 가져와 병합.

3여러 커밋을 하나의 커밋으로 압축해 병합하는 것은?

정답 A — merge --squash 후 git commit으로 하나의 커밋이 됨.

4브랜치의 기반(base)을 옮겨 선형 이력을 만드는 명령어는?

정답 D — git rebase는 이력을 한 줄로 펴 줌(재작성).

5새 브랜치를 만들면서 동시에 전환하는 명령어는?

정답 B — switch의 -c가 “생성+전환”. git switch bug는 이미 있는 브랜치로만 전환.

6로컬 main을 원격에 처음 올리며 추적관계를 설정하는 명령은?

정답 C — -u(=--set-upstream)로 이후엔 git push만 쳐도 됨.
